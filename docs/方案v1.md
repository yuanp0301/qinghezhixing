# 知识可视化站点方案（v1）

## 1. 产品定位与范围

本项目定位为“知识可视化站点”，用于沉淀和分发可交互的知识内容（如互动动画、交互式课件、可视化讲解页面）。

当前 v1 阶段先落地一个菜单模块：

- 菜单名称：`互动动画管理`
- 当前能力：
  1. 上传 HTML 文件
  2. 原文件存放到 OSS，并将 OSS 存储信息写入数据库
  3. 根据规则生成唯一 `key`
  4. 提供带 `key` 的下载 URL
  5. 用户通过 URL 下载 OSS 文件（同一 `key` 仅允许成功使用一次）

产品扩展性约束：

- 站点采用“多菜单模块”设计，后续可新增其他知识可视化菜单（例如资源库、模板管理、统计分析等），不影响当前 `互动动画管理` 模块独立迭代。

说明：本文件为产品与技术方案文档，不包含业务代码实现。

## 2. 核心业务流程

1. 用户上传 HTML 文件。
2. 后端校验文件合法性后上传至 OSS。
3. 后端写入文件主表记录，拿到 `file_id`。
4. 后端按既定规则生成 `key`，回写到主表。
5. 后端返回下载 URL（例如 `/d/{key}`）。
6. 用户访问下载 URL：
   - 校验 `key` 是否合法、是否为系统生成、是否未使用。
   - 校验通过后允许下载 OSS 文件。
   - 下载成功后将 `key` 标记为已使用，并写入使用日志表。
7. 同一 `key` 再次访问时，返回“已失效/已使用”。

## 3. Key 生成规则（已确认）

### 3.1 规则定义

- 用户名规范化：
  - `username_normalized = trim(lowercase(username))`
- 拼接原文：
  - `raw = username_normalized + ":" + file_id`
- 哈希与编码：
  - 输入编码：UTF-8
  - 哈希算法：SHA-256
  - 输出格式：hex 全长 64 位小写字符串

即：

`key = sha256(raw).hexdigest().lower()`

### 3.2 设计意图

- 保证同一 `(username, file_id)` 组合可重复计算出同一个 `key`。
- 在 `file_id` 全局唯一前提下，`key` 全局唯一。
- 使用固定分隔符避免拼接边界歧义。

## 4. 下载策略（一次性 key）

下载请求到达 `/d/{key}` 时，按顺序执行：

1. 校验 `key` 格式（64 位小写十六进制）。
2. 查询主表，确认 `key` 存在且为系统记录。
3. 校验 `key` 状态为 `unused`（未使用）。
4. 校验通过后发放下载资格并返回文件（重定向或代理流式）。
5. 下载成功后：
   - 主表将 `key_status` 置为 `used`
   - 写入 `used_at`
   - 写入使用日志表
6. 再次访问同一 `key` 时返回拒绝。

## 5. 数据库设计（建议）

## 5.1 文件主表 `files`

- `id`：主键
- `file_id`：业务文件 ID（唯一）
- `username_original`：原始用户名（可选）
- `username_normalized`：规范化用户名
- `key`：下载 key（唯一索引）
- `key_status`：`unused` / `used` / `expired`
- `used_at`：首次成功下载时间（可空）
- `original_filename`：原始文件名
- `content_type`：MIME 类型
- `size_bytes`：文件大小
- `oss_bucket`：OSS bucket
- `oss_object_key`：OSS 对象路径
- `created_at`
- `updated_at`

索引建议：

- `UNIQUE(key)`
- `UNIQUE(file_id)`
- `INDEX(username_normalized, created_at)`
- `INDEX(key_status)`

## 5.2 使用日志表 `key_usage_logs`

- `id`：主键
- `key`
- `file_id`
- `used_by`：使用者标识（有登录可用 user_id，匿名可存访客标识）
- `used_at`
- `client_ip`（可选）
- `user_agent`（可选）
- `result`：`success` / `reject_invalid` / `reject_not_found` / `reject_already_used`
- `reason`：详细原因（可选）

索引建议：

- `INDEX(key, used_at)`
- `INDEX(file_id, used_at)`
- `INDEX(result, used_at)`

## 6. 接口草案（v1）

## 6.1 上传接口

- 方法：`POST /api/files/upload`
- 入参：
  - `username`
  - `file`（仅 HTML）
- 出参：
  - `file_id`
  - `key`
  - `download_url`（例如 `/d/{key}`）
  - `expires_at`（若启用有效期）

## 6.2 下载接口

- 方法：`GET /d/{key}`
- 处理：
  - 校验 key
  - 校验未使用
  - 返回文件下载
  - 写使用记录并把 key 标记为已使用

## 7. 并发与一致性要求

必须保证“校验未使用 + 标记已使用”是原子操作，避免同一 `key` 并发下被重复成功下载。

建议手段（任选其一）：

- 数据库事务 + 条件更新（推荐）
- 乐观锁版本号
- 行级锁（`SELECT ... FOR UPDATE`）

成功判定建议：

- 以“成功发放下载资格”作为 `used` 标记时机。
- 若下载链路异常中断，可记录失败日志并保持状态一致性策略（按业务决定是否回滚）。

## 8. 安全与风控建议（v1）

- 文件类型白名单：仅允许 `.html` / `text/html`（双重校验）。
- 文件大小限制（例如 10MB）。
- OSS 使用私有读，下载使用受控访问（签名 URL 或后端代理）。
- 下载接口限流，防止枚举 `key`。
- 对异常和拒绝请求写日志，便于审计。

## 9. 异常场景与返回语义

- `key` 格式不合法：返回参数错误。
- `key` 不存在：返回未找到。
- `key` 已使用：返回已失效/已使用。
- 对应 OSS 文件不存在：返回资源异常并记录错误日志。
- 数据库写日志失败：不影响主流程或降级重试（按稳定性策略定）。

## 10. 待确认项（进入开发前）

1. 是否要求用户登录（`used_by` 字段来源）。
2. 下载 URL 是否设置有效期（`expires_at`）。
3. 下载返回方式采用“302 到 OSS 签名 URL”还是“后端代理流式”。
4. 文件保留策略（永久保存或定期清理）。
5. 错误码规范（是否对接统一网关/前端错误体系）。

## 11. OSS 接入补充（基于阿里云官方文档）

本章节基于阿里云 OSS「按功能分类的 API 概览」整理，聚焦本项目 v1 的最小必需能力。

### 11.1 接入原则

- 生产环境优先使用 OSS SDK（而非手写 HTTP 请求）：
  - SDK 已封装签名、重试、并发等通用逻辑，可降低接入风险。
- 认证使用 AccessKey 体系，签名版本采用 V4（官方推荐）。
- 请求必须发往 Bucket 所在地域对应的 Endpoint，禁止跨地域误配。
- 上线前需确认 OSS 使用限制与计费策略，避免容量或调用量超限导致故障。

### 11.2 与本方案对应的 OSS 操作映射

上传链路（`POST /api/files/upload`）建议使用：

- `PutObject`：上传 HTML 文件对象（v1 主路径）。
- `HeadObject`：可选，用于上传后校验对象是否存在及元信息（大小、ETag）。

下载链路（`GET /d/{key}`）建议使用：

- `GetObject`：后端代理下载时直接读取对象内容。
- 预签名 URL（SDK 生成）：302 跳转下载时使用，避免暴露长期凭证。

可选运维操作（v1 可不启用）：

- `DeleteObject`：若后续有文件清理策略，可用于对象回收。
- 生命周期（Lifecycle）规则：若后续有自动过期需求，可在 Bucket 侧配置。

### 11.3 Bucket 与权限建议

- Bucket ACL 建议为 `private`（私有读）。
- 不建议将业务下载能力建立在“公共读 Bucket”上。
- 推荐使用“业务层校验 key + OSS 受控访问（签名 URL 或后端代理）”双层控制。

### 11.4 错误处理与可观测性

根据 OSS API 通用响应规则：

- 成功通常返回 `2xx`。
- 失败返回 `4xx/5xx`，响应中包含错误码（`Code`）和错误信息（`Message`）。

工程实践建议：

- 后端记录每次 OSS 请求的 `x-oss-request-id`，用于故障排查和工单定位。
- 对常见失败分类处理：
  - 鉴权失败（签名/凭证问题）
  - 对象不存在
  - 限流或服务端异常
- 应用层对 OSS 异常做统一错误翻译，避免将底层细节直接暴露给前端。

### 11.5 v1 实施清单（OSS 维度）

1. 确认 Bucket 所在地域与 Endpoint。
2. 配置最小权限 RAM 策略（仅允许本项目需要的 Object 操作）。
3. 接入 OSS SDK，并固定签名版本与超时/重试参数。
4. 上传接口完成 `PutObject` 与数据库写入联动。
5. 下载接口在 key 校验通过后执行 `GetObject` 或签名 URL 发放。
6. 接入日志字段（含 `x-oss-request-id`）并纳入监控告警。
